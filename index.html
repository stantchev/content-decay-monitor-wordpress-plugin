WP-Cron automation

Daily background scans ‚Äî no manual work

A transient-based offset lets each daily cron fire advance the scan window safely across large sites with thousands of posts.

STEP 01

Plugin Activates

Registers cdm_daily_scan WP-Cron event on activation.

STEP 02

Cron Fires

Once per day, retrieves the stored transient offset (defaults to 0 for a fresh cycle).

STEP 03

Batch Scanned

Scans up to 20 posts (configurable). Updates _cdm_score meta on each.

STEP 04

Offset Advances

Saves new offset to transient. Next fire resumes from there. Resets when all posts are covered.

FAQ

Frequently asked questions

Quick answers to common questions about Content Decay Monitor.

Does scanning affect site performance?

No. Scans are triggered manually via AJAX or by the background WP-Cron job. The scanner uses fields=ids queries to avoid loading full post objects until needed, and results are cached in post meta so no computation happens on the front end or on page load.

Can I scan custom post types beyond "post"?

Yes ‚Äî use the cdm_post_types filter to add any post type. See the Customization section above for an example.

What happens if I have thousands of posts?

The daily WP-Cron job scans in batches (default 20) and stores its offset in a transient. Each daily fire advances the window until all posts are covered, then the cycle resets ‚Äî no timeouts, no memory pressure.

Where is the score data stored?

In standard post meta under the keys _cdm_score (integer 0‚Äì100) and _cdm_last_scan (UTC datetime). No custom tables. Data is removed on plugin uninstall.

Can I change the decay thresholds?

Yes ‚Äî go to Tools ‚Üí Content Decay ‚Üí Settings to configure publish age threshold, modified age threshold, minimum word count, and batch size.

Is the plugin free?

Completely free and open source under GPL v2. No premium tier, no upsells, no external APIs required.

Where can I report bugs or request features?

Open an issue on GitHub. Pull requests following WordPress Coding Standards are welcome.
